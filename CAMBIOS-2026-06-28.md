# Cambios de sesión — 2026-06-28

Documento para el equipo (humano + Claude). Explica QUÉ se hizo y POR QUÉ, para
que cualquiera de los dos (y su Claude) entienda el nuevo estado del repo.

## Contexto / objetivo

Convertir el repo en un **pipeline colaborativo para 2 personas** que comparten
el mismo repo y las **mismas cuentas** (ElevenLabs, Higgsfield). Requisitos:
1. Que tras `git pull` funcione igual para ambos, sin configurar cosas por separado.
2. Poder **relevarse**: si uno deja un vídeo a medias, el otro continúa.
3. Sacar la config hardcodeada a un `.env` y automatizar el pipeline (antes era manual).

## El concepto que vertebra todo: RECETA vs. RESULTADO

- **Receta** (ligera, **va a git**): scripts, `videos.json`, y por vídeo
  `script.md` + `frames.json` (timings + prompts de imagen) + `state.json` (progreso).
  Es la *instrucción* de cómo hacer el vídeo.
- **Resultado** (pesado, **NO va a git**): `.mp3`, `.png`, `.mp4`. Es la *salida*.

Como las cuentas son compartidas, el resultado se **regenera/recupera** desde
ElevenLabs/Higgsfield a partir de la receta. El medio pesado no se sincroniza por
git (que es malísimo con binarios): se reproduce. Eso hace que el repo sea ligero
y que el relevo entre los dos funcione.

## Decisiones acordadas

| Tema | Decisión | Por qué |
|---|---|---|
| Secretos (API keys) | `.env` **local**, compartido a mano por canal seguro | Un secreto NUNCA va a git, ni en repo privado (se filtra y queda expuesto para siempre). |
| Medios pesados | No se versionan; se recuperan/regeneran de las cuentas | Git no es para binarios; las cuentas compartidas son la fuente de verdad. |
| Modelo de imágenes | Seguir con Higgsfield `nano_banana_pro` | Mismo estilo stickman, pipeline ya montado, y sale más barato que API directa. |

## Ficheros nuevos

| Fichero | Qué es | Principio |
|---|---|---|
| `scripts/config.py` | Config compartida: lee el `.env`, expone todo al resto | **Single source of truth** — un solo sitio para la verdad |
| `.env.example` | Plantilla de variables (sin claves) que SÍ va a git | Documenta qué hace falta sin exponer secretos |
| `videos.json` | Registro de vídeos (composición, naming, playback, rutas) | **Data-driven**: el motor es genérico, los datos fuera del código |
| `scripts/orchestrate.py` | Encadena `voice→whisper→inject→images→validate→render` + `state.json` | Automatiza lo mecánico; gates HITL en lo crítico |
| `scripts/validate_assets.py` | Verifica que existen audio + TODAS las imágenes | **Fail fast**: caza el problema antes de un render de 10 min |
| `requirements.txt` | Dependencias Python | Reproducibilidad del entorno |
| `SETUP.md` | Arranque + uso del orquestador | Onboarding del colaborador |
| `channel/.../<video>/frames.json` | Receta de cada vídeo (timings + prompts) | El medio se regenera desde aquí |
| `channel/.../<video>/state.json` | Progreso del pipeline (para el relevo) | Handoff entre los dos |

  para que el orquestador inyecte los FRAMES de forma segura (no a mano).
- `.gitignore` — añadido `!.env.example` (el patrón `.env.*` se lo tragaba) y `*.png/jpg`
  ya cubre los medios.
- `CLAUDE.md` — actualizado a la nueva estructura y comandos.

## Cómo se usa ahora (lo esencial)

```bash
# Primer arranque (cada uno, una vez):
pip install -r requirements.txt
cp .env.example .env        # rellena las 2 API keys
cd remotion && npm install && cd ..

# Ver dónde se quedó un vídeo (clave para el relevo):
python scripts/orchestrate.py --video sueno-stick --status

# Trabajar por pasos (recomendado en equipo):
python scripts/orchestrate.py --video sueno-stick --steps voice,whisper,inject
```
Flujo de relevo: trabajas → `state.json` se actualiza solo → commit+push (receta+estado,
no medios) → el otro hace `pull`, mira `--status` y sigue.
